# Infinite Scroll & Paging Architecture

This document explains how infinite scroll pagination is implemented in the Canopy Native Android app using **Paging 3**, **Room**, and **RemoteMediator** for an offline-first experience.

---

## High-Level Overview

The app uses the Android Paging 3 library to load surveys in pages as the user scrolls. Data flows through three layers:

```
API (Django REST) ──► Room Database ──► UI (RecyclerView)
       ▲                    ▲                  ▲
  RemoteMediator       PagingSource       PagingDataAdapter
```

- **RemoteMediator** fetches pages from the API and saves them into Room.
- **PagingSource** (auto-generated by Room) reads pages from the local database.
- **PagingDataAdapter** renders items in a RecyclerView with automatic diffing.

This means the database is always the single source of truth. The network layer only writes into Room, and the UI only reads from Room.

---

## Paging Configuration

Defined in `SurveyRepository.java` using constants from `Constants.java`:

| Parameter | Value | Purpose |
|-----------|-------|---------|
| `pageSize` | 5 | Items fetched per API request |
| `prefetchDistance` | 2 (`PAGE_SIZE / 2`) | How many items before the end of the list trigger the next page load |
| `initialLoadSize` | 10 (`PAGE_SIZE * 2`) | Items loaded on the very first fetch |
| `maxSize` | 50 (`PAGE_SIZE * 10`) | Maximum items held in memory at once |
| `enablePlaceholders` | false | No placeholder items for unloaded data |

## Three Paging Modes

The `SurveyViewModel` manages three distinct view modes, each with different paging behavior:

### 1. ALL — All Surveys (Remote Paginated)

- **RemoteMediator**: `SurveyRemoteMediator`
- **PagingSource**: `surveyDao.getSurveysPaged()` — all surveys ordered by date/time descending
- **Remote Key ID**: `"surveys_all"`
- Full infinite scroll with API-backed pagination.

### 2. MINE — My Surveys (Remote Paginated, Filtered by Observer)

- **RemoteMediator**: `SurveyRemoteMediatorFiltered`
- **PagingSource**: `surveyDao.getSurveysByObserverPaged(observerId)`
- **Remote Key ID**: `"surveys_observer_{observerId}"` — unique per user
- Each observer gets independent pagination state, so switching between users doesn't corrupt page tracking.

### 3. FILTERED — Local Filter (No Remote Pagination)

- **RemoteMediator**: None
- **PagingSource**: `surveyDao.getSurveysFilteredPaged(dynamicQuery)` — raw query built from filter criteria
- Pagination happens entirely on cached local data. The user must pull-to-refresh to get fresh data from the API.

### Mode Switching

The ViewModel uses `Transformations.switchMap()` on the `viewMode` LiveData. When the mode changes, a new `Pager` is created and the old flow is discarded:

```
viewMode changes → switchMap re-evaluates → new Pager emits PagingData → adapter.submitData()
```

If the user is in ALL mode but applies a filter, the mode automatically switches to FILTERED (local-only). Clearing the filter switches back to ALL (remote-paginated).

## Remote Key Tracking

### RemoteKeyEntity

Stored in the `remote_keys` table:

| Column | Type | Purpose |
|--------|------|---------|
| `keyId` | String (PK) | Identifies the pagination stream (e.g., `"surveys_all"`, `"surveys_observer_abc123"`) |
| `nextPage` | Integer | The next page number to fetch on APPEND. `null` means end of data. |
| `currentPage` | Integer | The last page that was successfully loaded |
| `createdAt` | Long | Timestamp used for cache expiration (1-hour TTL) |

### RemoteKeyDao

| Method | Purpose |
|--------|---------|
| `insert(remoteKey)` | Save or update pagination state (uses `OnConflictStrategy.REPLACE`) |
| `getRemoteKey(keyId)` | Retrieve current pagination state for a given stream |
| `deleteByKey(keyId)` | Clear pagination state on REFRESH |

## API Response Format

The Django REST API returns paginated responses in this shape (`SurveyListResponse`):

```json
{
  "count": 50,
  "next": "https://api.example.com/surveys/?page=2",
  "previous": null,
  "results": [ ...survey objects... ]
}
```

- `next` — URL of the next page. If `null` or empty, we've reached the end.
- `results` — The survey items for the current page.
- The `hasNext()` method checks whether `next` is non-null/non-empty.

## RemoteMediator Load Flow

Both `SurveyRemoteMediator` and `SurveyRemoteMediatorFiltered` follow the same structure. The only differences are the API call (filtered vs. unfiltered) and the remote key ID.

### Load Type Handling

| LoadType | Behavior |
|----------|----------|
| **REFRESH** | Fetches page 1. Deletes the old remote key first. Triggered on initial load or pull-to-refresh. |
| **APPEND** | Fetches `remoteKey.nextPage`. Triggered when the user scrolls near the bottom. Returns end-of-pagination if `nextPage` is null. |
| **PREPEND** | Not supported. Immediately returns `MediatorResult.Success(endOfPaginationReached = true)`. |

### Step-by-Step Load Sequence

```
1. Determine which page to fetch:
   ├── REFRESH → page 1 (delete old remote key)
   ├── APPEND  → remoteKey.nextPage (or page 1 if no key exists)
   └── PREPEND → return immediately (not supported)

2. Call the API:
   └── surveyApi.getSurveys(page, PAGE_SIZE)
       or surveyApi.getSurveysFiltered(page, PAGE_SIZE, filters)

3. Process the response (inside a Room transaction):
   ├── If REFRESH: delete old remote key for this stream
   ├── For each survey in results:
   │   └── dbHelper.saveSurveyWithSections(survey)
   │       (upserts the survey + its sections atomically)
   ├── Determine if end of pagination: !response.hasNext()
   ├── Calculate nextPage: endReached ? null : page + 1
   └── Insert new RemoteKeyEntity(keyId, nextPage, currentPage, now)

4. Return MediatorResult.Success(endOfPaginationReached)
```

### Cache Expiration

The `initializeFuture()` method checks the age of the remote key's `createdAt` timestamp:

- **Cache older than 1 hour** → `LAUNCH_INITIAL_REFRESH` (re-fetch from API)
- **Cache younger than 1 hour** → `SKIP_INITIAL_REFRESH` (use local data)

This means opening the app within an hour of the last fetch will show cached data instantly without a network call.

## Complete User Flow: First Open

```
1. MainActivity created
2. Observes surveyViewModel.getPagedSurveys() (mode = ALL)
3. Pager created with SurveyRemoteMediator + Room PagingSource
4. RemoteMediator.initializeFuture() checks cache
   → No remote key exists → LAUNCH_INITIAL_REFRESH
5. load(REFRESH) called → fetches page 1 from API
6. API returns 5 surveys + hasNext = true
7. Surveys saved to Room, RemoteKeyEntity created:
   { keyId: "surveys_all", nextPage: 2, currentPage: 1 }
8. Room PagingSource emits 5 items
9. SurveyAdapter renders 5 items in RecyclerView
10. Since initialLoadSize = 10, APPEND is triggered immediately
11. load(APPEND) → fetches page 2
12. 5 more surveys saved, remote key updated to nextPage: 3
13. RecyclerView now shows 10 items
```

## Complete User Flow: Scrolling Down

```
1. User scrolls → RecyclerView reaches prefetchDistance (2 items from end)
2. Paging library triggers APPEND on RemoteMediator
3. RemoteMediator reads remote key → nextPage = 3
4. API call: getSurveys(page=3, pageSize=5)
5. Response: 5 surveys, hasNext = true
6. Save to Room, update remote key → nextPage = 4
7. Room PagingSource emits next batch
8. SurveyAdapter appends 5 new items to RecyclerView
9. SurveyLoadStateAdapter shows loading spinner at bottom during fetch
10. Repeat until API returns hasNext = false
```

## Complete User Flow: Last Page

```
1. APPEND triggered → fetch page N
2. API returns results with hasNext = false (next = null)
3. RemoteMediator sets nextPage = null in RemoteKeyEntity
4. MediatorResult.Success(endOfPaginationReached = true)
5. No more APPEND loads triggered
6. SurveyLoadStateAdapter hides (no spinner, no error)
7. User sees the end of the list
```

## Complete User Flow: Pull to Refresh

```
1. User pulls down on SwipeRefreshLayout
2. surveyViewModel.forceRefresh() called → clears all remote keys
3. Pager invalidated → triggers REFRESH
4. load(REFRESH) → deletes remote key, fetches page 1
5. Fresh data written to Room
6. PagingSource re-emits from the beginning
7. RecyclerView scrolls to top with fresh data
8. SwipeRefreshLayout spinner dismissed
```

## UI Components

### SurveyAdapter

Extends `PagingDataAdapter<SurveyEntity, SurveyViewHolder>` with a `DiffUtil.ItemCallback`:

- `areItemsTheSame()` — compares by `surveyId`
- `areContentsTheSame()` — compares `surveyId`, `updatedAt`, and `syncStatus`

This ensures efficient UI updates: only changed items are re-bound.

### SurveyLoadStateAdapter

Displays footer states at the bottom of the list:

| State | UI |
|-------|-----|
| **Loading** | Progress spinner |
| **Error** | Error message + Retry button |
| **NotLoading** | Hidden (nothing shown) |

Connected via:
```java
recyclerView.setAdapter(adapter.withLoadStateFooter(new SurveyLoadStateAdapter(adapter::retry)));
```

### MainActivity LoadState Listener

Observes the combined load state to manage UI:

```
adapter.addLoadStateListener(loadStates -> {
    refresh = loadStates.getRefresh();

    // Show empty state view when refresh completes with 0 items
    // Show progress bar during initial REFRESH loading
    // Show error toast/snackbar on REFRESH error
    // Dismiss SwipeRefreshLayout spinner when not loading
});
```

## Multi-Mediator Coexistence

A key design decision: both `SurveyRemoteMediator` and `SurveyRemoteMediatorFiltered` write into the **same** `survey_entries` Room table. They coexist safely because:

1. **Separate remote key IDs** — `"surveys_all"` vs `"surveys_observer_{id}"` prevent page tracking conflicts.
2. **Survey data is NOT deleted on REFRESH** — Only the remote key is cleared. This allows surveys fetched by one mediator to be visible in another mode's PagingSource query.
3. **Upsert strategy** — `saveSurveyWithSections()` uses insert-or-update, so duplicate surveys from overlapping API responses don't cause conflicts.
